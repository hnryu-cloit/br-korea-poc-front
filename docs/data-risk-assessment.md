# BR Korea POC 데이터 회신 기준 리스크 정리

## 목적
- 본 문서는 데이터 제공처 회신 내용을 기준으로, 현재 [onepager.md](/home/hyeonjeong/projects/br-korea-poc-front/docs/onepager.md)의 요구사항을 구현할 때 발생할 수 있는 리스크를 정리한 문서다.
- `onepager.md` 자체는 수정하지 않고, 별도 리스크 문서로 판단 근거와 범위 조정 포인트를 남긴다.

## 전제
- 아래 리스크는 데이터 제공처 답변 기준으로 작성했다.
- 아직 추가 제공 예정 데이터가 있으므로, 일부 리스크는 추후 완화될 수 있다.
- 다만 현재 시점 기준으로는 POC 문구, 데모 시연 범위, 백엔드 계산 로직 설명을 보수적으로 잡는 것이 안전하다.

## 1. 생산관리 리스크

### 1-1. 5분 단위 재고 스냅샷 부재
- `onepager.md`에서는 `5분 단위 재고 스냅샷`, `현재 재고`, `1시간 후 예측`, `재고 소진 1시간 전 알림`을 전제로 한다.
- 회신 내용 기준으로는 5분 단위 재고 snapshot 데이터가 없고, 일단위 재고만 관리하고 있다.
- 따라서 현재 재고와 재고 변화는 `Daily 기초 재고 + 시간대별 상품별 매출 + 생산 등록 데이터`로 역추정해야 한다.

발생 가능 리스크:
- 화면상 `실시간 재고`처럼 보이더라도 실제로는 추정치가 포함된다.
- POS 반영 지연, 판매 누락, 생산 등록 시점 차이 등이 있으면 재고 오차가 커질 수 있다.
- 점주가 기대하는 “지금 남은 수량”과 시스템 표시값이 다를 경우 신뢰도 저하가 발생할 수 있다.

권장 대응:
- UI 문구를 `실시간 재고`보다 `추정 현재고`, `운영 추정 재고`처럼 낮추는 것이 안전하다.
- `마지막 갱신 시각`과 함께 `매출/생산 데이터 기반 추정` 문구를 병기하는 것이 좋다.

### 1-2. 1시간 후 예측 재고 정확도 리스크
- `1시간 후 예측`은 가능하더라도, 입력값 자체가 역추정 재고이므로 예측값은 추정 위에 추정을 쌓는 구조가 된다.
- 재고 스냅샷이 없으면 예측 오차의 원인을 분리하기 어렵다.

발생 가능 리스크:
- 예측 실패 시 모델 문제인지 원천 재고 문제인지 구분이 어렵다.
- 생산 필요 시점 알림이 조기/지연 발생할 수 있다.

권장 대응:
- POC 단계에서는 정밀 예측보다 `위험도 분류` 중심으로 표현하는 것이 현실적이다.
- 예: `1시간 후 위험`, `주의`, `안전` 등 구간형 안내 우선.

### 1-3. 생산 리드타임 고정값 사용
- 회신 내용상 생산 리드타임은 전 품목 `1시간`으로 설정 요청을 받았다.
- 실제로는 품목별, 시간대별, 매장별 리드타임 차이가 존재할 가능성이 높다.

발생 가능 리스크:
- SKU별 추천 생산 시점이 실제와 다르게 계산될 수 있다.
- 일부 품목은 늦게 추천되거나 너무 이르게 추천될 수 있다.
- 생산 알림 전환율이 떨어질 수 있다.

권장 대응:
- 현재 POC에서는 전 품목 공통 1시간 가정을 명시한다.
- 후속 단계에서 SKU별 리드타임 또는 공정 그룹별 리드타임 보정이 필요하다고 문서화한다.

### 1-4. Chance Loss 정의 및 금액 환산 리스크
- 회신 기준 Chance Loss 정의는 `정상적으로 수요가 존재하는 시간대에서 일정 시간 이상 연속으로 매출이 발생하지 않는 경우를 품절 상태로 정의`하는 방식이다.
- 이는 실제 재고 소진을 직접 관측한 것이 아니라, 매출 부재를 품절로 추정하는 방식이다.

발생 가능 리스크:
- 실제 품절이 아닌 저수요 시간대를 품절로 오인할 수 있다.
- 진열 지연, 직원 미등록, POS 반영 지연도 품절처럼 보일 수 있다.
- Chance Loss 금액 산출이 과대 또는 과소 추정될 수 있다.
- 폐기 데이터가 충분하지 않으면 `과잉 생산 회피`와 `기회 손실 감소`를 함께 최적화하기 어렵다.

권장 대응:
- 1차 POC에서는 `Chance Loss 금액`보다 `품절 추정 이벤트`와 `감소 가능 비율` 중심으로 표현하는 것이 안전하다.
- 금액 환산은 `참고 지표` 수준으로 두고, 계산 기준 문구를 반드시 노출해야 한다.

### 1-5. 재료 소진 및 품절 처리 연동 리스크
- onepager에는 `재료-상품 연결 로직`, `재고 기반 품절 처리 제안`, `배달앱 품절 연동 준비`가 포함돼 있다.
- 현재 회신에서는 수불 내역 3개월치 추가 제공 예정이라고 했으나, 아직 실제 확인 전이다.

발생 가능 리스크:
- 재료-상품 매핑이 충분하지 않으면 재료 소진 알림은 설명 수준에 머무를 수 있다.
- 채널 품절 처리 제안은 가능해도 자동화 연동은 어려울 수 있다.

권장 대응:
- 수불 내역 도착 전까지는 품절 처리 기능을 `제안형 UI`로만 유지한다.
- 자동 연동보다는 `권장 액션` 중심으로 설계하는 것이 안전하다.

## 2. 주문관리 리스크

### 2-1. 주문 이력 재수신 전까지 추천 로직 검증 제한
- 회신 내용상 주문 데이터를 포함한 전체 데이터를 다시 발송하겠다고 했다.
- 즉 현재 시점에는 주문 추천 3안의 실제 검증 기반이 충분하지 않을 수 있다.

발생 가능 리스크:
- `전일`, `전주 동요일`, `점주 반복 패턴` 3축 추천안 중 일부가 현재는 목업 수준으로 남을 수 있다.
- 추천 근거 수치와 실제 확정 주문 데이터를 연결해 설명하기 어렵다.

권장 대응:
- 주문 데이터 재수신 전에는 추천안 구조와 UI 플로우 중심으로 구현하고, 수치 정확성은 제한적으로 표현한다.

### 2-2. 특수일/명절 마스터 부재
- onepager는 `날씨, 행사, 특수일 예외 변수 반영`을 요구한다.
- 회신 내용상 명절 및 공휴일 기간에 대한 별도 마스터 데이터는 현재 없다.
- 대신 과거 캠페인 이력은 데이터에 남아 있다고 한다.

발생 가능 리스크:
- `특수일 보정`이 안정적인 기준 테이블이 아니라 캠페인 로그 해석에 의존하게 된다.
- 설/추석/공휴일 시즌 주문 추천의 재현성이 낮아질 수 있다.
- 행사와 공휴일 효과를 분리해서 설명하기 어렵다.

권장 대응:
- 1차 POC에서는 `특수일 반영` 대신 `캠페인 이력 반영` 수준으로 문구를 낮춘다.
- 명절/공휴일 마스터는 후속 데이터 확보 항목으로 별도 관리한다.

### 2-3. 조기 품절 보정 정밀도 리스크
- onepager는 `조기 품절 품목에 대한 익일 추가 주문 제안`을 포함한다.
- 하지만 생산/재고 자체가 역추정 기반이면 조기 품절 탐지도 추정 오차 영향을 받는다.

발생 가능 리스크:
- 실제 품절이 아닌 상황도 조기 품절로 잘못 분류할 수 있다.
- 익일 추가 주문 추천이 과잉 주문으로 이어질 수 있다.

권장 대응:
- 조기 품절 보정은 강한 추천보다 `검토 필요` 수준의 보조 신호로 시작하는 것이 안전하다.

## 3. 매출/손익 분석 리스크

### 3-1. 상품 마스터 및 원가 데이터 미확정
- onepager는 `순이익`, `손익분기점`, `상품별 수익성`, `실행 인사이트`를 핵심 가치로 둔다.
- 하지만 회신 내용상 상품 마스터와 원가 데이터는 내부 논의 후 회신 예정이다.

발생 가능 리스크:
- 순이익 계산이 실제 원가 기반이 아니라 추정 또는 일부 비용 기반으로 축소될 수 있다.
- 손익분기점 분석 정확도가 떨어질 수 있다.
- 상품별 수익성 비교가 불완전하거나 임시값에 의존할 수 있다.

권장 대응:
- 원가 데이터 확정 전까지는 `참고용 손익 추정`으로 명확히 표기한다.
- 데모 문구에서 `실제 정산값과 차이 발생 가능`을 숨기지 않는 것이 안전하다.

### 3-2. 비교군 분석 범위 미스매치
- onepager는 `동일 행사`, `직영점 대비`, `비교군 기준 명시`를 요구한다.
- 현재 회신은 `역세지역 기준` 매출 데이터를 보냈다는 수준이다.

발생 가능 리스크:
- 상권 평균 비교는 일부 가능하더라도, 직영점 비교는 보장되지 않는다.
- 동일 행사 기준 비교도 프로모션 상세와 매출 연결 구조가 명확하지 않으면 구현이 약해진다.
- 사용자가 기대하는 비교군과 실제 비교군이 다를 수 있다.

권장 대응:
- 1차 비교군은 `역세권 평균`처럼 명확한 한 가지 기준으로 제한하는 것이 좋다.
- `직영점 비교`, `동일 행사 비교`는 데이터 검증 후 후속 범위로 넘기는 것이 안전하다.

### 3-3. 프로모션 데이터 활용 난이도
- 회신에서는 프로모션 내역은 첨부파일로 확인하라고 했다.
- 프로모션 파일이 구조화되어 있어도, 실제 매출 데이터와 기간/점포/상품 수준으로 정합하게 연결되는지는 별도 검증이 필요하다.

발생 가능 리스크:
- 프로모션 효과를 단순 서술은 가능해도 계량적으로 설명하기 어려울 수 있다.
- AI 답변에서 프로모션을 근거로 제시하더라도 실제 수치 연결성이 약할 수 있다.

권장 대응:
- 프로모션은 초기에는 설명형 컨텍스트로 쓰고, 수치 비교 근거로는 제한적으로 사용하는 것이 좋다.

### 3-4. 매장 맞춤형 분석 품질 편차
- 회신 내용상 매장 평수는 마스터 등록 기준으로 제공됐고, 0인 점포는 미등록 점포다.
- 추가 수기 파일 존재 여부도 아직 확인 중이다.

발생 가능 리스크:
- 일부 점포는 점포 특성 기반 맞춤형 분석이 약해질 수 있다.
- 매장 맞춤형 AI 답변 품질이 점포별로 달라질 수 있다.

권장 대응:
- 평수 미등록 점포는 별도 fallback 로직을 둬야 한다.
- 예: 상권, 매출 패턴, 운영 이력 중심의 맞춤화로 대체.

## 4. 데이터 정합성 및 모델링 리스크

### 4-1. ERD/DDL 제공은 긍정적이나 실제 검증은 별도 필요
- ERD, 테이블 생성 DDL, 외래키 지정 자료가 제공됐다는 점은 긍정적이다.
- Join 기준 불명확성 리스크는 일부 완화됐다.

남아 있는 리스크:
- 문서상 FK와 실제 데이터 품질이 다를 수 있다.
- 누락 키, 중복 키, 비정상 레코드가 있으면 조인 실패가 발생할 수 있다.

권장 대응:
- 실제 파일 입수 후 샘플 조인 검증을 별도로 수행해야 한다.

### 4-2. 상품명 정규화는 마스터 기준이라고 했지만 마스터 미확정
- 회신에서는 상품명이 상품 마스터 기준으로 동일하게 관리된다고 했다.
- 다만 상품 마스터 자체가 아직 미제공 또는 미확정이면 실제로는 검증이 끝난 상태가 아니다.

발생 가능 리스크:
- 상품명은 같아 보여도 상품 코드 체계가 중간에 바뀌었을 수 있다.
- 생산, 매출, 주문, 폐기 데이터를 상품 단위로 완전히 연결하지 못할 수 있다.

권장 대응:
- 상품 마스터 확보 전까지는 코드 기반 조인 가능 여부를 최우선 검증 항목으로 둔다.

## 5. 현재 시점 기준 종합 판단

### 구현 가능성이 높은 영역
- 생산관리 기본 흐름
  - 단, `실시간`이 아니라 `역추정 기반 운영 보조` 성격이 강함
- 주문관리 기본 UI
  - 추천 3안 구조, 점주 선택 저장, 근거 설명 흐름

### 구현은 가능하지만 표현 조정이 필요한 영역
- 1시간 후 예측 재고
- 생산 필요 시점 자동 감지
- Chance Loss 감소 효과
- 조기 품절 보정
- 프로모션/캠페인 반영 추천

### 현재 데이터 기준 리스크가 큰 영역
- 실제 원가 기반 순이익 계산
- 정교한 손익분기점 분석
- 직영점/동일 행사 비교
- 매장 평수 포함 맞춤형 비교 분석

## 6. 권장 범위 조정안

- 생산
  - `실시간 재고` 대신 `추정 현재고`
  - `Chance Loss 금액`보다 `품절 추정 리스크`와 `감소 가능 비율` 우선
  - 품절 처리 자동화보다 `제안형 액션` 우선

- 주문
  - `특수일 반영` 대신 `캠페인/과거 패턴 반영`
  - 조기 품절 보정은 강한 추천보다 `검토 보조 신호`로 운영

- 손익
  - 원가 데이터 확정 전까지 `참고용 손익 추정` 문구 명시
  - 비교군은 우선 `역세권 평균` 정도로 한정
  - `직영점 비교`, `동일 행사 비교`는 후속 범위로 분리

## 7. 한 줄 결론
- 현재 회신 기준으로는 `생산 추정 로직`은 POC 가능, `주문 추천`은 제한적 가능, `손익/비교분석`은 핵심 데이터 미확정으로 인해 onepager 기대 수준 대비 리스크가 가장 크다.
